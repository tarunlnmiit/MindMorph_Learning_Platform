# MindMorph Learning Platform — QA / End-to-End Test Plan

**Audience:** QA / Testing team
**Scope:** Everything implemented to date — the multi-agent learning prototype, the adaptive
learning loop, and the individual-agent test harness.
**Build under test:** `main` branch (latest). UI is a Streamlit app (`app.py`).
**Last updated:** 2026-06-13

---

## 1. What this product does (one-paragraph orientation)

MindMorph turns a learning goal (e.g. *"I want to learn Python for data science"*) into an
**interactive skill roadmap**. A team of LLM agents researches the goal, builds a **skill dependency
graph**, and lets the learner click any skill to get a **lesson** (teaching content + a practice
exercise). The learner's exercise is **graded live**, the score **updates mastery** on the graph, and
a low score **adapts the graph** — adding remedial prerequisite skills and regenerating the lesson to
target the learner's gaps. There are also two "quick" routes (a standalone lesson, a standalone
exercise) and a developer mode to test each agent in isolation.

**A note on non-determinism:** all content, lessons, and exercises are **generated by an LLM at
runtime**. The same query will produce different wording, different exercises, and different unit
tests on each run. Test cases below therefore assert on **structure and behavior** (a graph appears,
a node turns green, a remedial node is added) — **not** on exact text. Where an exercise must be
solved, read the on-screen **Grading harness** to see what is required, then use the recipes in
**Appendix A**.

## 2. Scope

### In scope (implemented — test these)
| Area | Feature |
|---|---|
| Full Orchestration → **SCOUT** route | Skill dependency graph generation, reviewer verdict, roadmap detail tabs (Academic / Market / Practical) |
| **Adaptive learning loop** | Node picker, composed lesson (content + exercise), lesson caching, live grading, mastery capture, graph adaptation (remedial nodes / unlock edges), score-aware regeneration, sticky mastery |
| Full Orchestration → **CONTENT** route | Standalone "quick lesson" (dual-path: creative + live web grounding) |
| Full Orchestration → **EXERCISE** route | Standalone exercise + live grading |
| **Individual Agent Test** mode | Orchestrator, Scout, Academic, Market, Practical agents in isolation |
| Cross-cutting | Logging, error handling, re-initialize, secrets validation |
| Automated tests | `pytest` suite |

### Out of scope (not implemented — do **not** raise bugs for these)
Persistence/accounts (everything is in-memory per browser session), user login/auth, Postgres/Redis,
RAG/vector search, multi-model routing, Next.js frontend, containerized grading sandbox, visual/diagram
generator, real dataset ingestion (dataset exercises return links). See `docs/IMPLEMENTATION_STATUS.md`
for the full roadmap.

## 3. Test environment & setup

### 3.1 Prerequisites
- macOS or Linux, **Anaconda/Miniconda** installed.
- A **`conda` env named `mindmorph`** (the project's required env).
- API keys:
  - **`GROQ_API_KEY`** — **mandatory** (every LLM call). Without it the app fails to start.
  - `APIFY_API_TOKEN` — *optional*; only the **Market** tab / Market agent needs it. Its absence is
    expected to degrade gracefully (no crash).
  - `GITHUB_TOKEN` — *optional*; improves the Practical agent's GitHub results.

### 3.2 One-time setup
```bash
# from the project root
git checkout main && git pull
conda run -n mindmorph pip install -r requirements.txt   # if env not already provisioned
```
Create a `.env` file in the project root:
```
GROQ_API_KEY=<key>
APIFY_API_TOKEN=<optional key>
GITHUB_TOKEN=<optional token>
```

### 3.3 Launch the app
```bash
conda run -n mindmorph streamlit run app.py
```
Browser opens at **http://localhost:8501**. Keep the terminal visible — runtime logs print there and
also to `logs/mindmorph.log`.

### 3.4 Run the automated test suite (separate terminal)
```bash
conda run -n mindmorph python -m pytest -q
```

## 4. How the three "routes" work (important for testers)

In **Full Orchestration**, an Orchestrator agent reads your query and routes it to one of three flows.
Phrase the query to force the route you want to test:

| Route | Triggered by (intent) | Example queries |
|---|---|---|
| **SCOUT** (full roadmap + adaptive loop) | "I want to learn / teach me / roadmap for …" | `I want to learn Python for data science` · `Teach me machine learning` |
| **CONTENT** (one quick lesson) | "Explain / give me a lesson on / what is …" | `Explain Python list comprehensions` · `Give me a quick lesson on REST APIs` |
| **EXERCISE** (one standalone exercise) | "Give me an exercise / a coding challenge on …" | `Give me a coding exercise on Python loops` |

> Routing is LLM-driven. If a query lands on the wrong route, rephrase using the cues above. A
> mis-route on an ambiguous query is **low severity**, not a blocker.

### Suite C — Adaptive learning loop (core feature)

Precondition: a roadmap is generated (Suite B).

| ID | Title | Steps | Expected | Sev |
|---|---|---|---|---|
| C1 | Picker ordering | Open the **"Pick a skill to study"** dropdown | Skills listed **foundational → intermediate → advanced** (not random) | Medium |
| C2 | Open lesson | Pick a node → **📖 Open lesson** | Spinner "Composing lesson…", then a lesson: heading, teaching content (markdown), and a **🏋️ Practice Exercise** at the end | Critical |
| C3 | Lesson caching | Open node X, switch to node Y, switch back to X | X loads **instantly** (no spinner) — served from cache | High |
| C4 | Grading harness display | Expand **"Grading harness"** on a coding exercise | Unit tests shown as **one coherent code block**; caption shows the **real test count** (number of `def test`s) | High |
| C5 | Grade — CORRECT | Submit a fully-correct solution (Appendix A) → **Grade my submission** | Green `N/N tests passed — score 100%`; "Adapting…" spinner. If the node has **no unmastered prerequisites**, it turns **✅** and `🎯` increments. If a prerequisite is still incomplete, it turns **🔒** and `🎯` does **not** increment — this is correct (see C14), not a bug | Critical |
| C6 | Adaptation on mastery | After C5 | An **unlock edge** is added from the mastered node to a downstream node (visible in the graph / JSON); **no** new node | Medium |
| C14 | Prerequisite-gated completion | Fail a node (→ spawns a remedial **prerequisite**), then **master that same node** while the new prerequisite is untouched | Node shows **🔒** (passed its own exercise but a prerequisite is incomplete), **not** ✅; `🎯` count does not include it. Then master the prerequisite → the node flips to **✅** and `🎯` increments | High |
| C15 | Locked lesson not openable | In the picker, pick a node shown with a **🔒** prefix (a prerequisite isn't complete) → **📖 Open lesson** | No lesson opens; a warning **"🔒 Locked — first complete: \<prerequisite names\>"** lists what to finish. A node downstream of a locked node is also locked | High |
| C16 | Unlock on prerequisite completion | Master the prerequisite(s) named in C15, then reopen the previously-locked node | The 🔒 prefix is gone; the lesson now opens normally | Medium |
| C7 | Grade — WRONG | On an **unmastered** node, submit a failing solution (Appendix A) | Red `0/N — score 0%`; "Adapting…" spinner; node turns **🔁**; a **new remedial node** appears with an edge pointing **into** the failed node | Critical |
| C8 | Score-aware regeneration | After C7, reopen the **same** failed node | Lesson **regenerates** (spinner) and now leads with a **"REMEDIAL FOCUS"** section targeting the gaps | High |
| C9 | Open remedial node | Pick the newly-added remedial node → Open lesson | A lesson for that prerequisite sub-skill renders | Medium |
| C10 | Grade — PARTIAL | On an **unmastered** node, submit a partially-correct solution (Appendix A), tuned to 50–79% | Mixed result (e.g. `3/5 — 60%`); node turns **▶ in_progress**; no remedial node | High |
| C11 | Sticky mastery | On a node already **✅**, submit a wrong solution | Node **stays ✅** (mastery is sticky); `best_score` unchanged in JSON; `attempts` increments | High |
| C12 | Stale panel cleared | Fail a node (C7), reopen it (new exercise generated) | The old grade result is **gone** — the panel does not show a result from the previous exercise | Medium |
| C13 | ID stability | Trigger several adaptations | Existing node ids are **never renamed/removed**; only new nodes/edges are added; previously-mastered nodes keep their ✅ | High |

> **Status glyphs reference:** ✅ complete · 🔒 prerequisites pending · ▶ in_progress (50–79) ·
> 🔁 needs_review (< 50). A node earns the **green ✅ only when it AND all its prerequisites
> (transitively) are mastered**; a node that passed its own exercise (score ≥ 80) but still has an
> incomplete prerequisite shows **🔒**. Per-exercise score→band on a 5-test exercise: 5/5 or 4/5 →
> mastered, 3/5 → ▶, ≤2/5 → 🔁. The `🎯 N / M` header counts **fully-complete** nodes.

## 6. Cross-cutting checks

- **Logging:** terminal + `logs/mindmorph.log` should show structured lines for each stage, e.g.
  `UI: generate requested…`, `Grader: coding result X/Y…`, `UI: adapting graph after grade…`,
  `Adaptation: +N node(s) +M edge(s)…`, `ContentAgent: generating … (remediation=True)`. No secrets
  (API keys) should ever appear in logs.
- **Security note (do not "fix" as a bug):** code grading runs the learner's Python in a subprocess
  with a wall-clock timeout and secret-scrubbed env. It is a **hang-guard for a local single-user
  prototype, not a security sandbox** (documented in `tools/code_executor.py`). Real isolation is a
  future hosted-deployment item. Do **not** run hostile code against a shared/production host.

## Appendix A — Exercise solution recipes

Exercises are LLM-generated, so the **function name, signature, and unit tests differ every run**.
The reliable procedure:

1. Expand the **Grading harness**. Read `from solution import <NAME>` and each `def test_…` assert.
2. Your editor content = the full `solution.py`. Define `<NAME>` to match.
3. Pick the outcome you want to test:

**CORRECT (→ 100% → ✅ mastered):** implement `<NAME>` so every assert passes. Watch for edge cases
the tests demand — e.g. raising `ValueError` on negative input, **with the exact message** the test
checks (`assert str(e) == '…'`).

**WRONG (→ 0% → 🔁 needs_review):** define the name but return a value that fails every assert. Universal:
```python
def <NAME>(*args, **kwargs):
    return None
```

**PARTIAL (→ 50–79% → ▶ in_progress):** implement the core logic correctly but skip the edge/validation
cases, so some tests pass and some fail. Then **read `passed/total`** and tune:
- landed ≥ 80% → break one more case;
- landed < 50% → fix one case.

**Worked example.** For a harness like:
```python
from solution import calculate_rectangle_properties
def test_normal():            # (5,3) -> {'area':15,'perimeter':16}
def test_zero_length():       # (0,3) -> {'area':0,'perimeter':6}
def test_negative_length():   # (-5,3) -> raises ValueError('Length and width must be non-negative')
```
- Correct (100%):
  ```python
  def calculate_rectangle_properties(length, width):
      if length < 0 or width < 0:
          raise ValueError('Length and width must be non-negative')
      return {'area': length * width, 'perimeter': 2 * (length + width)}
  ```
- Partial (passes the positive/zero tests, fails the negative ones):
  ```python
  def calculate_rectangle_properties(length, width):
      return {'area': length * width, 'perimeter': 2 * (length + width)}
  ```
- Wrong (0%):
  ```python
  def calculate_rectangle_properties(length, width):
      return {}
  ```

> Tip: the worked-example numbers printed in the **statement** are sometimes themselves
> LLM-miscalculated. Trust the **unit tests in the Grading harness**, not the prose examples.
